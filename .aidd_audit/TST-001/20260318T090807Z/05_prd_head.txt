Вопрос 1 (Blocker): Какие конкретные API/эндпоинты считаются "критическими" для RBAC-контроля?
Зачем: Определяет scope работы и какие middleware/декораторы нужно реализовать
Варианты: A) Только админские эндпоинты (/admin/*) B) Все мутационные API (POST/PUT/DELETE) C) Список из N конкретных эндпоинтов D) Все API с уровнем доступа > public
Default: B) Все мутационные API (POST/PUT/DELETE)

- Need to add audit logging for role operations and 403 responses
- Frontend role guards must sync with session updates without page refresh
- Ticket: TST-001

- All critical APIs must enforce RBAC with 403 on violations
- Role operations must be auditable
- Frontend must reflect role changes without refresh

## AIDD:RISKS
- Performance impact of live RBAC checks → caching strategy with TTL
- Session desync between backend/frontend → websocket/polling mechanism
- Audit log storage growth → retention policy and archiving

## AIDD:DECISIONS
- Live RBAC enforcement required per FS-RBAC-03 spec → compliance requirement
- Frontend sync without refresh → UX requirement for seamless role updates
- **Q1**: Critical APIs = все мутационные API (POST/PUT/DELETE)
- **Q2**: Текущая реализация = кастомная middleware на JWT-claims
- **Q3**: Frontend = React + Context/Redux
- **Q4**: Audit logs = база данных (отдельная таблица)
- **Q5**: Session sync = WebSocket (push-обновления)

## AIDD:GOALS
- Enforce live RBAC across critical APIs
- Add auditable role operations trail
- Synchronize frontend role guards with live session updates

## AIDD:NON_GOALS
- UI for role management (assumes existing admin interface)
- Migration of historical access logs
- Multi-tenant RBAC scope

## AIDD:ACCEPTANCE
- AC-1: Critical APIs return 403 for unauthorized role access
- AC-2: All role operations are logged with timestamp, user, action
- AC-3: Frontend role guards update within 5 seconds of session change

## AIDD:METRICS
- RBAC check latency P95 → ≤ 10ms
- Frontend role sync delay → ≤ 5 seconds
- Audit log coverage → 100% of role operations
