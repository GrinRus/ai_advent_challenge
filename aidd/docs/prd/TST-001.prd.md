## AIDD:CONTEXT_PACK
- FS-RBAC-03 spec requires live RBAC enforcement on critical API endpoints
- Need to add audit logging for role operations and 403 responses
- Frontend role guards must sync with session updates without page refresh
- Ticket: TST-001

## AIDD:NON_NEGOTIABLES
- All critical APIs must enforce RBAC with 403 on violations
- Role operations must be auditable
- Frontend must reflect role changes without refresh

## AIDD:OPEN_QUESTIONS
None

## AIDD:RISKS
- Performance impact of live RBAC checks → caching strategy with TTL
- Session desync between backend/frontend → polling mechanism
- Audit log storage growth → retention policy and archiving

## AIDD:DECISIONS
- Live RBAC enforcement required per FS-RBAC-03 spec → compliance requirement
- Frontend sync without refresh → UX requirement for seamless role updates
- **Q1**: Critical APIs = все мутационные API (POST/PUT/DELETE)
- **Q2**: Текущая реализация = X-Profile-Key header-based auth (custom middleware)
- **Q3**: Frontend = React + Zustand (not Redux)
- **Q4**: Audit logs = база данных (отдельная таблица)
- **Q5**: Session sync = Polling (fallback approach, simpler to implement)

## AIDD:GOALS
- Enforce live RBAC across critical APIs
- Add auditable role operations trail
- Synchronize frontend role guards with live session updates

## AIDD:NON_GOALS
- UI for role management (assumes existing admin interface)
- Migration of historical access logs
- Multi-tenant RBAC scope

## AIDD:ACCEPTANCE
- AC-1: Critical APIs return 403 for unauthorized role access
- AC-2: All role operations are logged with timestamp, user, action
- AC-3: Frontend role guards update within 5 seconds of session change

## AIDD:METRICS
- RBAC check latency P95 → ≤ 10ms
- Frontend role sync delay → ≤ 5 seconds
- Audit log coverage → 100% of role operations

## AIDD:ROLL_OUT
- **Phase 1 (MVP)**: Backend RBAC middleware + audit logging

## 2. Контекст и проблемы
- **Текущая ситуация**: Система использует X-Profile-Key header-based auth middleware для RBAC, но live enforcement не реализован для всех критических API. Frontend на React + Zustand не получает live-обновлений при изменении ролей пользователя. Audit-логи ролевых операций отсутствуют.
- **Проблемы/гипотезы**:
  1. Нет единой точки контроля RBAC для всех мутационных API — потенциальные security gaps
  2. Frontend не синхронизируется с изменениями ролей в реальном времени — пользователи видят устаревшие permissions
  3. Отсутствие audit trail для ролевых операций — невозможно отследить кто и когда получил/потерял доступ
  4. 403-ответы не логируются — сложно анализировать попытки несанкционированного доступа
- **Затронутые сегменты**: Все аутентифицированные пользователи системы (100%), администраторы управляющие ролями

## 3. Цели и метрики успеха
- **North Star / основная метрика**: 100% мутационных API (POST/PUT/DELETE) защищены live RBAC enforcement с корректными 403-ответами
- **Поддерживающие метрики**:
  - RBAC check latency P95 → ≤ 10ms
  - Frontend role sync delay → ≤ 5 секунд
  - Audit log coverage → 100% ролевых операций
- **Контрольные метрики**:
  - Доля ошибок 500 при RBAC проверках → ≤ 0.1%

## 5. Пользовательские сценарии

### Сценарий 1: Администратор изменяет роль пользователя
1. Администратор вызывает API для изменения роли пользователя (PUT /api/users/{id}/role)
2. Система проверяет RBAC (только admin может менять роли) — возвращает 403 если нет прав
3. При успехе: роль обновляется в БД, операция записывается в audit_log
4. Frontend пользователя получает обновление при следующем polling-запросе (≤ 30s)
5. Zustand store обновляется и UI пересчитывает доступные действия без перезагрузки

### Сценарий 2: Пользователь пытается выполнить запрещенное действие
1. Пользователь с ролью "viewer" пытается вызвать POST /api/resources
2. RBAC middleware проверяет X-Profile-Key headers — определяет недостаточно прав
3. API возвращает 403 Forbidden
4. Попытка записывается в audit_log (user_id, endpoint, timestamp, denied)
5. Frontend получает 403 и показывает сообщение о недостаточных правах

### Сценарий 3: Сессия пользователя обновляется в реальном времени
1. Пользователь работает в приложении (React + Zustand)
2. Администратор отзывает роль через API
3. Frontend получает обновление при следующем polling-запросе
4. Zustand store обновляет auth state
5. UI скрывает элементы, требующие отозванной роли

## 6. Требования
### 6.1 Функциональные
- [ ] Все мутационные API (POST/PUT/DELETE) должны проходить RBAC проверку перед выполнением
- [ ] RBAC middleware должна интегрироваться с существующей X-Profile-Key header-based auth системой
- [ ] При отказе в доступе API возвращает 403 с информативным сообщением
- [ ] Audit-лог записывает: timestamp, user_id, action (GRANT/REVOKE/DENIED), target_resource, role_changes
- [ ] Polling endpoint для получения обновлений ролей пользователя (каждые 30 секунд)
- [ ] Frontend Zustand store обрабатывает polling-ответы и обновляет auth state
- [ ] Role guards на frontend блокируют UI-элементы при отсутствии permissions

### 6.2 Нефункциональные
- [ ] RBAC check latency P95 ≤ 10ms (использовать in-memory caching с TTL)
- [ ] Polling endpoint availability ≥ 99.5%
- [ ] Audit log storage: отдельная таблица в БД с retention policy
- [ ] Graceful degradation: при недоступности polling endpoint frontend продолжает работу с текущей сессией
- [ ] Безопасность: audit-логи не должны содержать sensitive data (пароли, токены)

### 6.3 Acceptance criteria
- [ ] AC-1: Все POST/PUT/DELETE API возвращают 403 для пользователей без необходимых ролей
- [ ] AC-2: Все операции изменения ролей (GRANT/REVOKE) логируются в audit_log таблицу
- [ ] AC-3: Frontend role guards обновляются в течение 30 секунд после изменения ролей в backend (polling interval)
- [ ] AC-4: RBAC check latency P95 ≤ 10ms под нагрузкой 1000 RPS

## 7. Ограничения и зависимости
- **Технические**:
  - Интеграция с существующей X-Profile-Key header-based auth middleware (нельзя ломать обратную совместимость)
  - Polling endpoint должен масштабироваться с текущей инфраструктурой
  - Audit-логи хранятся в существующей БД (отдельная таблица)
  - Frontend: React + Zustand (текущий стек)
- **Процессные**:
  - Требуется согласование с security team по формату audit-логов
  - Релиз должен быть zero-downtime deployment
- **Внешние**: Нет внешних зависимостей (все компоненты внутренние)
